# OrgBench パイロット実験レポート

**実験日**: 2026-03-12〜13
**研究目的**: LLMマルチエージェントシステムにおける情報流通設計パターンが集合的意思決定品質に与える影響の検証
**対象学会**: AAMAS 2027 / IEEE ICA 2027

---

## 1. 実験概要

### 1.1 研究課題（RQ）

| RQ | 問い |
|----|------|
| RQ1 | マルチエージェント構成はシングルエージェントに対して有意に高品質な出力を生成するか？ |
| RQ2 | 階層の深さ（flat vs deep）は出力品質に影響するか？ |
| RQ3 | 通信トポロジー（hub vs mesh）は出力品質に影響するか？ |
| RQ4 | レビュープロトコルの有無は出力品質に影響するか？ |
| RQ5 | モデル均質性（同一モデル vs 異種モデル）は出力品質に影響するか？ |

### 1.2 実験設計

- **独立変数**: 11種類の組織構成（情報流通設計パターン）
- **従属変数**: 6次元の品質スコア（LLM-as-Judge、1-5スケール）
- **タスク**: ビジネス提案書作成（5テーマ）
- **反復**: 5回 / 構成・テーマ
- **総実行数**: 11構成 × 5テーマ × 5反復 = **275ラン**
- **Judge評価**: 275ラン × 2回独立評価 = **550評価**

## 3. 主要知見

### 3.1 RQ1: マルチ vs シングルエージェント

**結論: マルチエージェント構成は全次元でシングルエージェントを大幅に上回る（d=1.23〜1.81）**

- single_agent (1.82) は全構成中最低で、最良のマルチ構成 flat_mesh (3.30) との差は1.48ポイント
- Cohen's d = 1.58（overall_quality）は非常に大きな効果量
- 特にfinancial_rigor (d=1.81) とnovelty (d=1.81) で差が顕著
- シングルエージェントは検索ツールを使えるにもかかわらず、専門的な分析深度で劣る

### 3.2 RQ2: 階層の深さ（flat vs deep）

**結論: 浅い階層（flat）が深い階層（deep）より優位**

- flat_mesh (3.30) > anchor (2.92), deep_mesh (2.84)
- flat_hub (3.22) > anchor (2.92)
- 階層が深いと情報の伝達ロスが生じ、最終出力の品質が低下する可能性

### 3.3 RQ3: 通信トポロジー（hub vs mesh）

**結論: mesh通信はhub通信と同等以上**

- flat_mesh (3.30) ≥ flat_hub (3.22)
- deep_mesh (2.84) ≈ anchor (2.92, hub)
- matrix_hub (2.98, hub+mesh混合) > anchor (2.92)
- メッシュ通信によるピア間情報共有は品質向上に寄与するが、階層深度ほどの影響はない

### 3.4 RQ4: レビュープロトコル

**結論: レビューありがレビューなしより優位**

- anchor (2.92, レビューあり) > deep_noreview (2.68, レビューなし)
- 差は0.24ポイントで、他の要因（階層、通信）ほど大きくはない

### 3.5 RQ5: モデル均質性

**結論: 高性能モデル統一 >> 低コストモデル統一 ≈ 異種構成**

- homo_haiku (3.22, 全Haiku) ≈ flat_hub (3.22, 異種)
- homo_gpt (2.56, 全GPT-4o-mini) < anchor (2.92, 異種)
- Haikuは高コスト ($3.659) だが品質は最上位
- GPT-4o-mini統一は最もコスト効率が良い ($0.213) が品質は低い
- **コスト効率最良はflat_hub**: $0.814で overall 3.22を達成

## 4. 操作チェック（Manipulation Check）

構成間でメッセージフローが設計通りに異なることを確認:

| 構成 | 平均メッセージ数 | 備考 |
|------|-----------------|------|
| anchor/deep系 | 14.0 | 6エージェント+レビュー |
| flat系 | 12.0 | 浅い階層で少ない |
| deep_noreview | 10.0 | レビュー工程なし |
| single_agent | 1.9 | 1エージェントのみ |

→ 設計意図通りの差異が確認され、独立変数の操作が有効に機能している。

## 5. 制約と脅威

### 5.1 内的妥当性への脅威
- **LLM-as-Judge のバイアス**: 評価者がGemini Flashに限定。モデル固有の選好が結果に影響する可能性
- **プロンプト設計**: 各エージェントのシステムプロンプトが結果に大きく影響。最適化の余地あり
- **温度パラメータ**: 全構成で温度0.7を使用。タスクによっては最適でない可能性

### 5.2 外的妥当性への脅威
- **タスク限定**: ビジネス提案書作成のみ。他のタスクドメインへの一般化は検証が必要
- **モデル依存**: 特定のモデル（Haiku, GPT-4o-mini, Gemini Flash）に依存。モデル世代が変われば結果が変わる可能性
- **テーマ数**: 5テーマは限定的。より多様なテーマでの検証が望ましい

### 5.3 構成妥当性への脅威
- **overall_quality の解釈**: 6次元の重み付けが暗黙的にJudgeに委ねられている

## 6. 実験実行の技術的詳細

### 6.1 実行環境
- **プラットフォーム**: WSL2 (Ubuntu) on Windows
- **Python**: 3.12
- **主要ライブラリ**: litellm (統一LLM API), tavily (Web検索)
- **実行時間**: 約6時間（断続的実行、ネットワーク断による中断・再実行含む）

### 6.2 発生した問題と対応

| 問題 | 原因 | 対応 |
|------|------|------|
| Anthropic tool_calls形式エラー | litellm→Anthropic変換時のメッセージ形式不一致 | raw_tool_callsの保存・再利用 |
| Anthropic tools=必須エラー | 履歴にtool_use存在時にtools省略不可 | 全ラウンドでtools指定 |
| Judge全スコア3.0 | max_tokens=1024でJSON出力が途切れ | max_tokens=4096に増加 |
| ネットワーク断 | 移動中のDNS解決失敗 | idempotent skip機能で再実行 |
| Tavily API制限 | 検索クレジット消費 | クレジットチャージ後に再実行 |

### 6.3 コスト内訳
- **LLMエージェント実行**: $11.58
- **LLM-as-Judge評価**: Gemini Flash使用（低コスト）
- **Web検索（Tavily）**: 別途
- **総予算**: ¥10,000 (~$67) のうち約$12使用

### B. 効果量の解釈ガイド

- **η² (イータ二乗)**: 全分散のうち構成間差異が説明する割合。0.14以上で大きな効果。
- **Cohen's d**: 2群間の標準化された平均差。0.8以上で大きな効果。
- 本実験では5/6次元でη² > 0.14（large）、全次元でd > 0.8（large）。
