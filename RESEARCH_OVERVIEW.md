# Research Overview: The Science of AI Agent Organization

> **一言**: AIエージェントの「組織の作り方」を、実証・理論・適応の3段階で科学する研究
>
> 最終更新: 2026-03-14
> 著者: Koh Shinoda

---

## 1. 研究の全体ストーリー

### 1.1 問題意識

2025-2026年、LLMベースのマルチエージェントシステム（MetaGPT、ChatDev、CrewAI、AutoGen等）が急速に普及している。これらのシステムは「CEO」「エンジニア」「レビュアー」といった組織的メタファーを採用し、エージェント間のメッセージ経路・権限配分・レビュー構成を暗黙に設計している。

**しかし、その「組織の作り方」が出力品質にどう影響するかは、誰も体系的に調べていない。**

- MetaGPTは瀧型パイプラインを採用 → 「これがベスト」と主張
- ChatDevは階層型チャットチェーンを採用 → 「これがベスト」と主張
- CrewAIは階層/逐次を選べる → どちらが良いかは開発者任せ

各フレームワークが自分の構造を推すだけで、**同一条件での公平な比較は一度もされていない**。

### 1.2 なぜLLMエージェントで組織を研究するのか

人間の組織論（Galbraith, Mintzberg, Lawrence & Lorsch等）は60年の歴史があるが、**人間組織では制御実験が不可能**。文化・政治・感情・暗黙知・動機づけが交絡し、「組織構造の純粋な効果」を測れない。

LLMエージェントにはこれらの交絡因子がない。残るのは**情報フロー設計パターン**——メッセージ経路、権限配分、レビュー構成——の純粋な効果のみ。

→ LLMエージェント組織は、**組織の情報処理効果を史上初めて制御実験できる系**である。

### 1.3 3本の論文の関係

```
論文0 (Phase 0): 事実の発見    →「組織の作り方で、AIの出力品質は変わるのか？」
    ↓ 変わる。ではなぜ？
論文1 (Phase 1): 因果の説明    →「情報理論で説明できるか？」
    ↓ 説明できる。では改善できるか？
論文2 (Phase 2): 問題の解決    →「AIの組織は自分で最適構造に変われるか？」
```

**発見 → 説明 → 解決** という自然科学の王道的進行。

## 2. 論文0: OrgBench — 事実の発見

### 2.1 概要

| 項目 | 内容 |
|------|------|
| タイトル | OrgBench: How Information Flow Design Patterns Affect Collective Decision Quality in LLM Multi-Agent Systems |
| 状態 | パイロット実験完了（275回）、論文ドラフトv0.1 |
| ターゲット | AAMAS 2027 / IEEE ICA 2027 |
| コード | `orgbench/` ディレクトリ |
| 設計書・分析 | `Phase0/` ディレクトリ |

### 2.2 何をしたか

**OrgBench**というベンチマークフレームワークを構築し、LLMマルチエージェントシステムの**情報フロー設計パターン**（階層深度・通信トポロジー・レビュープロトコル・モデル構成・エージェント数）を独立変数として制御実験を行った。

- **11構成**（1シングルエージェント + 10マルチエージェント）
- **5テーマ**のビジネス提案書作成タスク
- **5反復** × 構成 × テーマ = **275実行**
- **LLM-as-Judge**による6次元品質評価（550評価）
- 使用モデル: CEO=Claude Haiku 4.5, 実務=GPT-4o-mini, 調査=Gemini 2.5 Flash
- **総コスト: ~$12**

### 2.3 何が分かったか

| 知見 | 効果量 | 意味 |
|------|--------|------|
| マルチエージェント >> シングルエージェント | d = 1.23〜1.81 | 組織化すること自体に大きな価値がある |
| 浅い階層 > 深い階層 | Δ = 0.38〜0.46 | 中間管理層は情報ロスを生む |
| メッシュ通信 ≥ ハブ通信 | 限定的 | ピア間共有は有効だが効果は小さい |
| レビューあり > レビューなし | Δ = 0.24 | 品質管理には効果があるが最大要因ではない |
| 高性能モデル統一 >> 低性能モデル統一 | 顕著 | モデル品質は構造設計と同等以上に重要 |
| 全設計変数の効果量 | η² = 0.12〜0.45 | **情報フロー設計は第一級の設計関心事** |

### 2.4 ベスト構成

| 構成 | overall_quality | コスト | 特徴 |
|------|----------------|--------|------|
| flat_mesh | **3.30** | $0.999 | 浅い階層 + メッシュ通信（品質最良） |
| flat_hub | **3.22** | $0.814 | 浅い階層 + ハブ通信（コスパ最良） |
| homo_haiku | **3.22** | $3.659 | 全Haiku（高品質だが高コスト） |
| single_agent | **1.82** | $0.905 | 明確に最下位 |

### 2.5 この論文で答えられない問い

- **「なぜ」深い階層が品質を落とすのか？** → 論文1で回答
- **「いつ」どの構造が最適なのか？（タスク依存性）** → 論文0の拡張実験 or 論文1で部分回答
- **「動的に」構造を変えられるか？** → 論文2で回答

## 3. 論文1 (Phase 1): Information Theory — 因果の説明

### 3.1 概要

| 項目 | 内容 |
|------|------|
| タイトル案 | The Information-Theoretic Cost of Hierarchy: Modeling LLM Multi-Agent Organizations as Communication Networks |
| 状態 | 計画段階 |
| ターゲット | NeurIPS 2027 / ICML 2027 / AAMAS 2027 |
| コード | `orgbench/src/orgbench/info_theory/` (予定) |
| 設計書 | `Phase1/` ディレクトリ |

### 3.2 核心的問い

> **なぜ深い階層は品質を落とすのか？ 最適なトポロジーの理論的根拠は何か？**

論文0は「浅い方が良い」という事実を示したが、因果メカニズムは不明。情報理論で説明する。

### 3.3 理論的アプローチ

**エージェント組織を情報処理グラフ G = (V, E, C) としてモデル化する。**

- 各エージェント = 有限容量の**通信チャネル**（入力を受け取り、処理し、出力する）
- メッセージの中継 = チャネルの直列接続
- **Data Processing Inequality (DPI)**: チャネルを通るたびに相互情報量は単調に減少する
  - I(X; Yₙ) ≤ I(X; Yₙ₋₁) ≤ ... ≤ I(X; Y₁)
  - 深い階層 = 多段中継 = 情報損失の蓄積 → **これが「深い階層が悪い」理由の数理的説明**

ただしLLMは単純なノイズチャネルではない。要約・推論により情報を圧縮しつつ新情報を付加しうる。DPIがLLMチャネルにどの程度当てはまるかは実証的問い。

### 3.4 測定するもの

| 指標 | 何を測るか | 計算方法 |
|------|-----------|---------|
| Semantic Similarity Decay | 各ステップで元情報からどれだけ意味が離れるか | embedding間のコサイン類似度の減衰率 |
| Entity Preservation Rate | 固有名詞・数値などの具体情報がどれだけ生き残るか | NER抽出エンティティの残存率 |
| Numerical Fidelity | 数値データが正確に引き継がれるか | 入力数値の最終出力での正確な引用率 |
| Compression Ratio | 情報がどれだけ圧縮されるか | 入力トークン数 / 出力トークン数 |
| Mutual Information Estimate | 入出力間の情報量 | MINE等の推定手法 |

### 3.5 実験計画

1. **既存データ再分析** (コスト$0): パイロット275回の全メッセージログから情報指標を計算
2. **追加実験** (~$100):
   - chain長を2/3/4/5/6と変化させて情報歪みの階層依存性を測定
   - star/chain/mesh/treeで同一タスクを実行してトポロジー依存性を測定
   - **Controlled Information Injection**: 入力にトレーサー情報（架空の統計値等）を埋め込み、各ステップでの生存率を追跡

### 3.6 期待される成果

- **Information Distortion Curve**: 階層深度 d と情報損失の定量的関係（線形/指数/対数のどれか）
- **トポロジー別の情報保存効率**: mesh > star > chain の理論的根拠
- **レビューゲートのフィードバック効果**: Shannon (1956) のフィードバック付きチャネル理論との対応
- **LLMチャネルの特殊性**: DPIに従わないケース（情報付加）の同定と条件

### 3.7 なぜこれがトップ会議に通るか

- **理論的novelty**: LLMエージェントネットワークを情報理論で定式化した研究は存在しない
- **理論 + 実証の両輪**: 数理モデル → 予測 → 実験検証のサイクル
- **汎用性**: LLMの世代が変わっても情報理論的フレームワーク自体は陳腐化しない
- **現在の研究との接続**: 論文0のOrgBenchデータを理論的に再解釈するため、新規データ収集の負担が小さい

## 4. 論文2 (Phase 2): Dynamic Organizational Adaptation — 問題の解決

### 4.1 概要

| 項目 | 内容 |
|------|------|
| タイトル案 | Self-Reconfiguring Agent Organizations: Information-Theoretic Triggers for Dynamic Structural Adaptation in LLM Multi-Agent Systems |
| 状態 | 計画段階（Phase 1完了後に着手） |
| ターゲット | ICLR 2028 / NeurIPS 2027 Workshop |
| コード | `orgbench/src/orgbench/adaptive/` (予定) |
| 設計書 | `Phase2/` ディレクトリ |

### 4.2 核心的問い

> **LLMエージェント組織は、タスクに応じて自分で構造を変えられるか？ そして固定構造を上回れるか？**

現実の人間組織はadaptive。プロジェクトの局面に応じてチーム構成を変え、権限を委譲し、コミュニケーション経路を組み替える。しかし現在のマルチエージェントシステムは**全て固定構造**。

### 4.3 Phase 1（情報理論）との接続

Phase 1で確立する情報理論的フレームワークが、動的適応の**理論的基盤**になる:

| 問い | Phase 1が提供する答え |
|------|---------------------|
| **いつ**適応すべきか？ | information processing demandが現構造のchannel capacityを超えたとき |
| **何に**適応すべきか？ | information distortionを最小化するトポロジーに |
| **効果を**どう測るか？ | 適応前後のinformation preservation rateの変化で |

Phase 1なしでは適応トリガーも適応先も場当たり的になる。Phase 1があれば**理論駆動の適応アルゴリズム**を設計できる。

### 4.4 適応メカニズム

**適応トリガー**（いつ構造を変えるか）:

| トリガー | 検出方法 | アクション |
|---------|---------|-----------|
| Information Bottleneck | 中心ノードのinputがcontext windowの80%超 | hierarchy flattening |
| Information Starvation | 末端エージェントの出力が入力の20%未満 | 直接通信チャネル追加 |
| Redundancy Overload | 同一情報が3経路以上で重複 | hub-and-spoke切替 |
| Quality Plateau | レビュースコアが2ステップ改善なし | role reassignment |
| Task Complexity Spike | サブタスクの出力長が想定の2倍超 | agent spawn |

**適応アクション**（何を変えるか）:
- Agent Spawn: 新エージェントの動的追加
- Role Reassignment: system promptの動的変更
- Hierarchy Flattening/Deepening: 報告構造の動的変更
- Communication Rewiring: 通信トポロジーの動的変更

**メタコントローラー**:
- Option 1 (主): Rule-Based — Phase 1の理論から導出した閾値に基づく決定木
- Option 2 (副): LLM-Based — メタエージェントが情報フロー状態を分析して適応判断

### 4.5 実験計画

- **ベースライン**: OrgBench最良固定構造、Single-Agent-Strong、Random Adaptation、Oracle Adaptation
- **タスク**: 途中で複雑度が変化するシナリオ（complexity escalation、domain shift、multi-phase）
- **評価**: 既存6次元品質 + Adaptation Frequency/Latency/Effectiveness/Overhead/Net Benefit

### 4.6 なぜこれがトップ会議に通るか

- **実用的インパクト**: 全てのマルチエージェントフレームワークに適用可能な設計原理
- **理論的裏付け**: Phase 1の情報理論に基づく適応（場当たり的でない）
- **novelty**: LLMエージェント組織の自己再構成は未開拓領域
- **明確なベースライン**: Phase 0のOrgBenchが強力な固定構造ベースラインを提供


### 5.2 設計思想

- **組織構造のYAML外部化**: コード変更なしで構造を変数化できる
- **オーケストレータ駆動**: エージェントが自律的に動くのではなく、オーケストレータがフローを制御（再現性のため）
- **マルチプロバイダ対応**: litellm経由でClaude/GPT/Geminiを統一的に利用
- **全ログ記録**: 全メッセージ・全LLM呼び出しを保存（事後分析可能）

## 9. 「あなた何の研究をしているの？」への回答

### 30秒版

AIエージェントを複数組み合わせて仕事をさせるとき、「誰が誰に報告し、誰がレビューし、誰が最終判断するか」という**組織の作り方**が出力品質に大きく影響することを実験で発見しました。今は、**なぜ**浅い組織が深い組織より良いかを情報理論で説明し、最終的にはAIが**自分で最適な組織構造に変わる**仕組みを作ろうとしています。

### 3分版

LLMベースのマルチエージェントシステムが普及していますが、エージェント間の「組織設計」——階層、通信経路、レビュー体制——は各フレームワークが勘で決めており、体系的な比較研究がありませんでした。

私はOrgBenchというベンチマークを作り、11種類の組織構造を同一条件で比較しました。275回の実験の結果、**浅い階層がη²=0.21で有意に深い階層を上回る**こと、**マルチエージェントはd=1.2-1.8でシングルエージェントを大幅に上回る**ことなどが分かりました。

次のステップとして、この「なぜ」を情報理論で説明します。各エージェントを通信チャネルとしてモデル化し、Data Processing Inequalityに基づいて**階層が深いほど情報が失われる**ことを数理的に示し、実験データで検証します。

最終的には、この理論を使って**タスク実行中にエージェント組織が自動的に構造を最適化する**仕組みを構築します。情報ボトルネックを検知したら階層を平坦化し、情報不足を検知したら直接通信を開くといった、理論駆動の適応アルゴリズムです。

**インパクト**: AIエージェントを使う全ての企業・開発者に対して、「このタスクにはこの組織構造が最適で、タスクが変わったらこう変えるべき」というエビデンスベースの設計指針を、理論的根拠付きで提供します。
